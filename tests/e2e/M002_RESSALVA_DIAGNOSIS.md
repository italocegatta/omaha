## Root cause

The `_login_and_select_italo` helper in
`tests/e2e/test_s04_user_journey.py:187-196` logs in as user `family`
with the test password:

```python
page.fill(SELECTORS["login_user"], "family")
page.fill(SELECTORS["login_pass"], "test-password")
page.click(SELECTORS["login_submit"])
page.wait_for_url(re.compile(r"/profiles$"))
```

`family` was the shared single-user account before commit
`35bf15d feat(phase-02): palette contrast fixes, multi-user seed, openspec infra`
(2026-06-16 11:49) split the seed into per-user accounts `Italo` + `Ana`
(see `src/omaha/seed.py:25-27`):

```python
DEFAULT_USERS: tuple[tuple[str, int], ...] = (
    ("Italo", 0),
    ("Ana", 1),
)
```

The e2e test helpers were not updated to match. The session-scoped
`live_url` fixture in `tests/e2e/conftest.py:73-128` still calls
`alembic upgrade head` + `seed()` correctly — that part still produces
the right DB — but the user-facing login form rejects `family` as
"Usuário ou senha inválidos." (verified in browser, screenshot
`/tmp/s06_e2e_debug/post_login_family.png`).

- **Reason:** Same stale `family` user as S06.

### S05 test 2 — `test_class_color_tokens_defined_in_root`

- **Fails at:** line 333.
- **Reason:** Real test bug, separate from the login issue. The
  test asserts every `--class-N` token starts with `#`:

  ```python
  assert v.startswith("#"), f"--class-{k} not a hex color: {v!r}"
  ```

  Commit `35bf15d feat(phase-02): palette contrast fixes` deliberately
  converted `--class-4` and `--class-6` to `oklch()` syntax for
  better contrast (per the Phase 2 audit). The test was never
  updated to accept the new format. The browser **does** resolve
  `oklch(0.53 0.13 50)` correctly as a color — the assertion is
  checking string format, not visual correctness.

- **Production code:** correct. `--class-4` is
  `oklch(0.53 0.13 50)` (5.16:1 contrast vs ink — passes WCAG AA).
  `--class-6` is `oklch(0.52 0.10 200)` (4.89:1).
- **Fix:** change the assertion to accept any non-empty CSS color
  (e.g. drop the `startswith("#")` check; the empty-string check
  on the line above is the real invariant). Belongs in the same
  helper-fixer change as the login fix, but it's a separate
  one-line tweak.

## Non-e2e suite findings (added after running `pytest tests/ --ignore=tests/e2e`)

Tally: **83 failed, 12 errors, 213 passed, 3 skipped** — **before AND
after this change**, confirmed by `git stash` + re-run (same numbers

### Cause: same `family` user bug, scope larger than expected

The stale-`family` assumption is not limited to `tests/e2e/`. It
propagates into TestClient-style tests too. First symptom caught in
the output:

```
tests/test_t04_e2e.py::test_full_login_profile_dashboard_logout_flow
    assert 200 == 303
    login_response = client.post("/login", data={"username": "family", ...})
```

`test_t04_e2e.py` is a TestClient integration test (despite the
"e2e" filename) — it does not launch a browser, it hits the app
in-process. Same `data={"username": "family", ...}` payload, same
rejection. Most `tests/test_t03_imports_routes.py` ERRORs are setup
failures cascading from this same login not redirecting (the
session never gets `user_id`, so subsequent requests redirect
back to /login).

### Scope of recommended fix (expanded)

`fix-stale-test-user-after-multi-user-seed` should sweep both:

1. `tests/test_t04_e2e.py` and any other TestClient test that
   posts to `/login` with `username=family` — grep for `"family"`
   in `tests/` will find them all.
2. `tests/e2e/test_s04_user_journey.py:190` — `family` → `Italo`.
3. `tests/e2e/test_s05_user_journey.py:333` — drop `#` assertion.

### Why this change does not (and should not) fix it

Per the original design: this change's task 6.1 is to "verify
the unit/integration suite is green". It is not — but it never
was, on this host, with this seed. The failure is not caused by
the conftest change (conftest lives in `tests/e2e/`, doesn't
affect the non-e2e collection). Adding the fix to this change
would violate the "atomic change" principle and would re-introduce
the (also-stale) `family` assumption into a different test file
the way the original `35bf15d` migration missed them.

### Deviations from the original proposal / design

1. **Drop `assert len(v) in (4, 7)`** at
   `tests/e2e/test_s05_user_journey.py:334` in addition to the
   `startswith("#")` on line 333. The length check is also
   hex-specific (`oklch(0.53 0.13 50)` has length 19) and would
   have remained a failure even after the design's planned
   `startswith` removal. Documented in §8.3.
2. **Helpers got an optional `username` parameter** in
   `test_t02_classes_routes.py`, `test_t03_imports_routes.py`,
   `test_s04_t02_import_commit.py`. The structural drift
   (one-user-with-two-profiles → two-users-with-one-profile-each)
   means cross-profile tests must re-authenticate as the profile's
   owner. The helper now defaults `username` to the profile name
   so existing call sites stay Italo-logged-in. Used by 3 tests.
3. **`test_t03_auth.py:128` "Ana profile" test** was rewritten
   to log in as Ana (her profile id is 2) and select her
   profile. The original selected profile 2 as Italo, which
   404s post-`35bf15d` (cross-profile ownership check).
4. **`tests/test_t03_auth.py:101` and `tests/test_t04_e2e.py:85`**
   drop the `assert "Ana" in profiles_page.text` check. The
   per-user picker shows only the logged-in user's profile;
   Italo's picker has 1 button, not 2.
5. **`tests/test_t02_seed.py:140` profile-ownership assertion**
   rewritten. Each profile is owned by its namesake user
   (Italo→Italo, Ana→Ana) — they don't all share `user_id`.
