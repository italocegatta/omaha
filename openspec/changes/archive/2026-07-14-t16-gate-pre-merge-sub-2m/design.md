## Context

The test suite has 6 execution lanes with wildly different wall-clock costs:

| Lane | Task | Baseline | Marker |
|------|------|----------|--------|
| unit | `test-unit` | 15.37s | `-m unit` |
| integration | `test-integration` | 190.12s | `-m integration` |
| audit integration | `test-audit-integration` | 25.04s | path-based |
| e2e | `test-e2e` | 192.50s | path-based (Playwright) |
| bdd | `test-bdd` | 176.87s | path-based (pytest-bdd) |
| visual | `test-visual` | 55.36s | path-based (Playwright) |

The existing `task check` (`lint && test-unit`) already runs in ~30s but:
1. Not formally named as a "gate" вЂ” unclear intent.
2. Not documented in PERFORMANCE.md as a lane definition.
3. No explicit timing target documented.

## Goals / Non-Goals

**Goals:**

- Define a formal `gate-fast` task that runs lint + unit in < 2 min wall-clock.
- Document the lane split: fast gate (lint + unit) vs. full suite (everything).
- Deprecate `check` in favor of `gate-fast` (clearer intent).
- Update PERFORMANCE.md with gate definitions and timing expectations.

**Non-Goals:**

- No test optimization or parallelization.
- No CI pipeline YAML changes.
- No coverage reporting changes (T13).
- No integration test splitting.
- No new test markers or conftest changes.

## Decisions

### D-T16.1 вЂ” `gate-fast` = lint + test-unit (serial)

Command: `uv run prek run --all-files && uv run pytest -m unit --cov=src/omaha --cov-report=xml:reports/coverage.xml`

This matches the existing `check` task content. The lint step runs prek (ruff format check, ruff --fix, hygiene hooks). The test-unit step runs pytest with the `unit` marker and coverage XML output.

Rationale: Serial execution keeps the task simple and debuggable. Lint failures surface before test noise. Total wall-clock ~30s leaves 90s of headroom under the 2 min target.

Alternatives considered:

- **Parallel lint + unit** вЂ” requires backgrounding one process and collecting exit codes. Complexity not justified for a 30s task. Revisit if lint time grows past 60s.
- **Include integration** вЂ” integration baseline is 190s, blows the 2 min budget. Rejected.
- **Include spec verification** вЂ” `openspec validate` adds 5-10s. Could be a `gate-spec` variant. Not in scope for the fast gate.

### D-T16.2 вЂ” Deprecate `check`, add `gate-fast`

`task check` becomes a deprecated alias: `task check` в†’ prints deprecation warning в†’ runs `gate-fast`. Or simply: `check` stays as-is, `gate-fast` is the new canonical name documented in PERFORMANCE.md and AGENTS.md.

Rationale: "check" is generic. "gate-fast" communicates intent (pre-merge gate, fast lane). Keeping `check` as alias avoids breaking muscle memory.

Alternatives considered:

- **Rename `check` to `gate-fast`** вЂ” breaks anyone using `task check`. Keep both.
- **Add `gate-full` for the full suite** вЂ” `task test` already exists. Redundant.

### D-T16.3 вЂ” Lane documentation in PERFORMANCE.md

Add a "Gate definitions" section after the existing "Lanes de execuГ§ГЈo" section. Content:

```
## Gate definitions

### Fast gate (pre-merge) вЂ” target < 2 min

Command: `uv run task gate-fast`

Runs: lint (prek) + unit tests (pytest -m unit)
Excludes: integration, audit integration, e2e, bdd, visual, coverage reporting

Baseline (2026-07-13): ~30s wall-clock (lint ~15s + unit ~15s)

Use when: merging to main, pre-push check, quick feedback loop.

### Full suite вЂ” no timing target

Command: `uv run task test`

Runs: unit + integration + audit integration + e2e + bdd + visual
Baseline (2026-07-13): ~10+ min wall-clock

Use when: release validation, full regression sweep.

### Browser lanes вЂ” run async

Commands: `task test-e2e`, `task test-bdd`, `task test-visual`
These require Playwright + Chromium. Run in CI or overnight.
Not part of the fast gate.
```

## Risks / Trade-offs

- **Risk:** `gate-fast` name not discoverable if user only knows `check`. в†’ **Mitigation:** Keep `check` as alias. Document both in PERFORMANCE.md.
- **Risk:** Lint time grows as codebase grows, pushing gate-fast past 2 min. в†’ **Mitigation:** Current lint ~15s, 4x headroom. Monitor.
- **Trade-off:** Coverage XML is still generated by `gate-fast` (via `test-unit` addopts). This adds ~1-2s overhead. в†’ **Acceptable:** the XML is useful for CI artifact upload and doesn't meaningfully impact the 2 min target.

## Migration Plan

1. Add `gate-fast` task to `pyproject.toml` `[tool.taskipy.tasks]`.
2. Update `check` help text to note deprecation in favor of `gate-fast`.
3. Update `tests/PERFORMANCE.md` with gate definitions section.
4. Run `task gate-fast` to verify < 2 min wall-clock.

Rollback: remove `gate-fast` task, revert PERFORMANCE.md. Trivial.
